# Guardian Engineering Framework (GEF)

> A software engineering framework that transforms workflow rules into automated tools. It guarantees traceability, security, and quality on every project, from the very first commit.
> 
> **🛡️ Industry Standards Enforced:** GitHub Flow (mandatory PRs), OWASP Security Limits (Rate Limiting, JWT Exp), Adaptive Clean Code Metrics (Startup / Standard / Mission Critical), and **AI Anti-Bypass Guarantee** (Crash Clause, complete `.cursorrules`, Intention CI).
>
> *Guardian — the mechanical guardian that protects your projects from silent drift.*

## 1. Philosophy

GEF is built on a single principle: **engineering rules should not be re-read — they must be mechanically enforced.**

- The [`ENGINEERING_PLAYBOOK.md`](./ENGINEERING_PLAYBOOK.md) is the absolute source of truth. It defines universal rules (Git traceability, documentation, architecture, security, TDD, ADR, Kanban). It never contains project-specific information.
- The [`PROJECT_CONFIG.template.md`](./PROJECT_CONFIG.template.md) is the project-specific complement (cloud, database, milestones). It is automatically generated by the generator and must be completed by the project owner.
- **Nothing in this repository is project-specific.** The GEF is universal and agnostic.

## 3. Installation and Usage

GEF is designed to be used directly without needing to clone the repository, just like `create-next-app` or `create-vite`.

**Prerequisites:** Node.js (v18+), Git, GitHub CLI (`gh`) for Kanban features.

### Available Commands

| Command | Description |
|---|---|
| `npx create-gef` | Launches the interactive generator and creates a new project |
| `npx create-gef update` | Updates the Playbook, Prompts and Hooks in an existing project |
| `npx create-gef --help` | Displays help and all available commands |
| `npx create-gef --version` | Displays the current framework version |

### Create a New Project

```bash
npx create-gef
```

### Update an Existing Project

From the root of an existing GEF-generated project, update the Playbook, prompts and Git hooks to the latest framework version:

```bash
npx create-gef update
```

### Display Help

```bash
npx create-gef --help
npx create-gef --version
```

### Local Framework Development

If you are modifying the GEF framework itself and want to test the CLI locally:

```bash
# 1. Clone the repository
git clone https://github.com/Gnzikoune/GEF.git GEF
cd GEF

# 2. Install dependencies
npm install

# 3. Make the local command globally accessible
npm link
```

## 4. The Project Generator (Block A)

### What the Generator Does

The assistant asks **11 questions**, then automatically executes:

| Step | Action |
|---|---|
| **1. Scaffolding** | Installs the chosen framework (`npx create-next-app`, `npm create vite`, etc.) |
| **2. GEF File Structure** | Creates the Diatáxis structure: `docs/tutorials/`, `docs/how-to/`, `docs/reference/`, `docs/explanation/adr/` |
| **3. ADR Template** | Creates `docs/explanation/adr/0000-template.md` ready to use |
| **4. Configuration** | Generates `PROJECT_CONFIG.md` pre-filled with all choices (stack, cloud, DB, git, lint, severity, language) |
| **5. Documentation** | Generates `README.md` and `docs/research/RESEARCH_LOG.md` |
| **6. Linter** | Generates `biome.json`, `.eslintrc.json`, or `ruff.toml` based on the choice |
| **7. Docker** | Generates `docker/Dockerfile` and `docker/docker-compose.yml`. If PostgreSQL: creates `database/init.sql` mounted in the container |
| **8. Playbook & AI Prompts** | Copies Playbook and Prompts to `.gef/` **injecting the Hard Limits adapted** to the chosen severity level |
| **9. Git Hooks** | Generates dynamic hooks (`pre-push` blocks `main` if GitHub Flow, runs tests if TBD) |
| **10. CI/CD** | Generates `.github/workflows/main.yml` adapted to the stack and cloud provider |
| **11. Release Please** | Generates `.github/workflows/release-please.yml` to automate tags and releases |

### Supported Stacks

| Framework | Scaffolding | Docker | CI |
|---|---|---|---|
| Next.js (React) | `npx create-next-app@latest` | Multi-stage → Node | Setup Node 20 + lint + tests |
| React (Vite) | `npm create vite@latest` | Multi-stage → Nginx | Setup Node 20 + lint + tests |
| API Node.js (Express) | `npm init` + `express` | Node Alpine + DB service | Setup Node 20 + lint + tests |
| API Python (FastAPI) | `venv` + `requirements.txt` | Python 3.12-slim + DB service | Setup Python 3.12 + flake8 + pytest |
| Empty project | — | Generic Alpine | Generic |

### Supported Git Strategies

| Strategy | `pre-push` hook behavior |
|---|---|
| **GitHub Flow** *(Recommended)* | Blocks any `git push` attempt to `main`. Forces branch and Pull Request usage. |
| **Trunk-Based Development** | Allows pushes to `main`, but runs local tests before validating. |

### Severity Levels (Hard Limits)

The chosen level is injected into the Playbook and AI Prompts generated in `.gef/`. A "Mission Critical" project's AI will **never** generate a function longer than 15 lines.

| Level | Max functions | Max params | Max complexity | Max JSON payload |
|---|---|---|---|---|
| **Startup / R&D** | 50 lines | 4 | 15 | 5 MB |
| **Standard / Enterprise** *(Recommended)* | 30 lines | 3 | 10 | 1 MB |
| **Mission Critical** | 15 lines | 2 | 5 | 100 KB |

### Supported Linters

| Linter | Generated file | Commands added in `package.json` |
|---|---|---|
| **Biome** | `biome.json` | `npm run lint`, `npm run lint:fix` |
| **ESLint + Prettier** | `.eslintrc.json` + `.prettierrc` | `npm run lint`, `npm run lint:fix` |
| **Ruff** *(Python)* | `ruff.toml` | — |
| **None** | — | — |

### Supported Cloud Providers

| Cloud | Effect |
|---|---|
| **Vercel** | Generates `vercel.json`, removes Docker question, auto-deployment in CI on push `main` |
| **AWS** | AWS deployment job in CI on tag `v*.*.*` |
| **GCP / Azure** | Automatic GitHub Release on tag `v*.*.*` |
| **None** | Automatic GitHub Release on tag `v*.*.*` |

### Supported Databases

| DB | Effect |
|---|---|
| **PostgreSQL** | PostgreSQL `db` service in `docker-compose.yml` |
| **MongoDB** | MongoDB `db` service in `docker-compose.yml` |
| **Supabase** | Generates `supabase/config.toml` + `supabase/migrations/` + `supabase/seed.sql` |
| **None** | No DB configuration |

## 5. Git Hooks (Block B)

Automatically installed by the generator in `.git/hooks/` of each project.

| Hook | Applied rule |
|---|---|
| **`commit-msg`** | Blocks any commit whose message does not follow the `Conventional Commits + Kanban reference` format. Format: `feat: description (#42)`. |
| **`pre-commit`** | Detects plaintext secrets (API keys, tokens). Checks formatting (linter). Analyzes file sizes based on chosen severity. **Blocks any direct commit on `main` or `master`.** |
| **`pre-push`** | **Dynamic**: Blocks any direct push to `main` if the project uses GitHub Flow. Runs local tests if using Trunk-Based Development. |

These hooks are generated on the fly by the generator based on team choices, and installed in `.git/hooks/` of the project.

To update hooks in an existing project:

```bash
npx create-gef update
```

## 6. The CI/CD Pipeline (Block C)

The generator creates two files in `.github/workflows/`:

**`main.yml` — Quality Control & Deployment**
- Adapted to your stack and cloud. Triggered on push `main`, `feat/**`, `fix/**`, tags `v*.*.*`, and pull requests.
- **Jobs:** setup runtime → install → lint → tests → security analysis → deployment (Vercel/AWS) or GitHub release.

**`release-please.yml` — Release Automation**
- On each push to `main`, automatically generates a Release Pull Request with the correct version number (calculated from your `feat:` and `fix:` commits) and the `CHANGELOG.md`.
- When you merge this PR: the Git tag and GitHub Release are created automatically.

## 8. The Virtual Tech Lead (Block E)

Beyond generation, GEF transforms AI into an **autonomous Tech Lead** through three rules inscribed in the Playbook:

### Kanban Steering & Pull Requests (§14)
The AI creates its own tickets (`gh issue create`), links each commit to a ticket (`feat: ... (#42)`), opens Pull Requests (`gh pr create`) and **asks for your validation before merging**.

### ADR Auto-Documentation (§15)
Before any major architectural choice (new dependency, new service), the AI **must** write a report in `docs/adr/` using the provided template. It cannot code without first documenting its decision.

### AI-Driven TDD (§16)
Before writing application code, the AI writes the E2E test (Playwright) describing the expected behavior. The code is then written to make this test pass.

> **Grandfather Clause (§0.5):** These rules apply to new code. The AI never proactively refactors old code to make it compliant, unless explicitly asked.

## 9. The AI Anti-Bypass Guarantee (Block F)

GEF goes beyond textual rules. It **mechanically enforces** best practices on AI from the moment the project is opened, without the user needing to repeat them.

### How it Works

| Mechanism | File | Effect |
|---|---|---|
| **Native IDE Rules** | `.cursorrules` / `.windsurfrules` | Any AI (Cursor, Windsurf, Copilot) reads these files at startup and instantly knows §0 to §10 of the Playbook (Clean Code, Architecture, OWASP Security, Git Flow, Tests, Workflows). |
| **Crash Clause** | `prompts/system_prompt.md` | The AI is instructed to stop immediately and report any obstacle, instead of silent improvisation. |
| **Pull Request Checklist** | `.github/PULL_REQUEST_TEMPLATE.md` | The AI (and humans) must physically check validations (Tests, Docs, ADR) before a PR can be merged. |
| **Linter Lock (Hard Limits)** | `biome.json` / `.eslintrc.json` / `ruff.toml` | The linter is configured with limits (e.g. 15 lines max, 2 params) and will crash if the AI attempts to bypass the framework. |
| **Local Lock** | `hooks/pre-commit` | A direct commit on `main` is physically impossible, as is a commit that doesn't pass the Linter. |
| **CI Validation (Intention & Tests)** | `ci-templates/pr-intention-check.yml` & `main.yml` | CI rejects PRs without business intent, and blocks if test coverage < 80%. |
| **Propagation** | `generator/features/scaffold-ai-rules.js` | Each generated project automatically inherits the complete `.cursorrules` (single source of truth). |

> The power lies here: the user never has to explain the rules to the AI. They are already there.
