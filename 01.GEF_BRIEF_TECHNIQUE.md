## 1. Contexte et objectif

Il existe déjà un référentiel documentaire — l'`ENGINEERING_PLAYBOOK.md` — qui définit des règles d'ingénierie universelles (traçabilité Git, documentation, architecture, sécurité, CI/CD, gestion des branches, séparation R&D/Production, etc.), applicables à tous les projets d'un même porteur, indépendamment du langage ou du domaine (voir Annexe 11.1).

**Le problème que ce brief résout :** ce Playbook est aujourd'hui un document passif. Il dépend de la discipline manuelle (humaine ou IA) pour être respecté. L'objectif est de le transformer en **framework exécutable** : un ensemble d'outils qui appliquent automatiquement les règles du Playbook, plutôt que de compter sur le fait qu'elles soient relues et suivies à chaque session de travail.

**Résultat attendu :** un dépôt nommé `GEF` (Gildas Engineering Framework), installable ou clonable dans n'importe quel nouveau projet, qui fournit :

- un générateur qui initialise un nouveau projet conforme au Playbook en une commande ;
- des hooks Git qui empêchent les commits non conformes ;
- un template CI/CD réutilisable qui applique les contrôles qualité automatiquement ;
- une bibliothèque de prompts qui garantit qu'une IA travaillant sur un projet applique le Playbook sans qu'on ait à le lui rappeler.

---

## 2. Principes directeurs (non négociables)

Ces principes s'appliquent à l'ensemble des briques ci-dessous. Toute décision d'implémentation qui les contredirait doit être remontée avant d'être prise.

- **Le Playbook reste la source de vérité.** Le framework l'applique, il ne le duplique pas. Les hooks, la CI et les prompts doivent référencer les règles du Playbook plutôt que de les recopier en dur, pour éviter toute divergence dans le temps.
- **Séparation stricte universel / spécifique.** Rien dans `GEF/` ne doit contenir d'information propre à un projet particulier (nom de bucket, provider cloud, nom de client). Ces informations vivent uniquement dans le `PROJECT_CONFIG.md` de chaque projet consommateur (voir Annexe 11.2).
- **Agnostique du langage autant que possible.** Le générateur et les hooks doivent fonctionner quel que soit le stack (Python, Node, etc.), avec des points d'extension pour la configuration spécifique à un langage (linter, formatteur, runner de tests), plutôt que de figer un seul écosystème.
- **Rien ne doit être silencieux.** Un hook qui bloque un commit, une étape CI qui échoue, doivent produire un message clair expliquant *quelle règle* du Playbook a été violée et *comment* corriger — jamais un simple code d'erreur.
- **Dégradation progressive acceptable.** Si un outil externe (linter, scanner de sécurité) n'est pas disponible dans l'environnement, le framework doit avertir et continuer plutôt que de bloquer totalement le travail, sauf pour les vérifications de sécurité critiques (secrets en clair), qui restent bloquantes.

## 3. Architecture générale du framework

```
GEF (dépôt indépendant)
   │
   │  cloné / installé dans chaque nouveau projet
   ▼
Projet consommateur
   ├── PROJECT_CONFIG.md          (spécifique, non fourni par GEF)
   ├── .gef/                      (config locale générée à partir de GEF)
   ├── .git/hooks/                (installés par GEF)
   └── .github/workflows/         (copiés/générés depuis GEF)
```

GEF n'est pas copié-collé manuellement à chaque fois : il expose une commande d'installation qui met en place ces éléments dans le projet cible.

## 4. Brique A — Générateur de projet (CLI)

### 4.1 Objectif

Une commande qui initialise un nouveau projet 100% conforme à la structure standard définie au §12 du Playbook, en une seule exécution.

### 4.2 Comportement attendu

Commande cible (le nom exact est laissé libre à l'implémentation, ex. `gef new` ou `./new-project`) :

```bash
gef new <nom-du-projet> [--lang python|node|other]
```

Elle doit :

1. Créer l'arborescence standard :
   ```
   docs/adr/
   docs/research/
   src/
   tests/
   scripts/
   .github/
   assets/
   infra/
   docker/
   database/
   README.md
   CHANGELOG.md
   PROJECT_CONFIG.md
   LICENSE
   .gitignore
   ```
2. Générer un `README.md` minimal avec les sections attendues par le Playbook §2 (Fonctionnalités, Installation, Architecture) — vides mais structurées, prêtes à remplir.
3. Générer un `PROJECT_CONFIG.md` à partir d'un template (voir Annexe 11.2), avec des placeholders explicites (`<À COMPLÉTER>`) plutôt que des valeurs inventées.
4. Générer un `RESEARCH_LOG.md` vide avec l'en-tête de format attendu (numérotation des entrées).
5. Générer un `.gitignore` adapté au langage choisi (`--lang`), incluant systématiquement : fichiers d'environnement (`.env`), dépendances locales, caches de build, environnements virtuels.
6. Initialiser le dépôt Git (`git init`) si ce n'est pas déjà fait, et appeler la routine d'installation des hooks (Brique B).
7. Copier le template CI/CD adapté (Brique C) dans `.github/workflows/`.
8. Afficher un résumé final listant ce qui a été créé et les étapes manuelles restantes (ex. « Compléter PROJECT_CONFIG.md avant le premier commit »).

### 4.3 Contraintes

- Ne jamais écraser un fichier existant sans confirmation explicite (idempotence prudente : si le dossier cible existe déjà et contient des fichiers, demander confirmation ou proposer un mode `--merge`).
- La commande doit être exécutable sans dépendance lourde (éviter d'imposer un runtime supplémentaire si le porteur ne l'utilise pas déjà — privilégier un script shell portable ou un script dans le langage le plus universellement disponible dans l'environnement cible).

### 4.4 Critères d'acceptation

- [ ] Exécuter la commande sur un dossier vide produit une arborescence strictement conforme au §12 du Playbook.
- [ ] Le `.gitignore` généré ne laisse passer aucun fichier sensible typique (`.env`, `node_modules/`, `__pycache__/`, `venv/`).
- [ ] Les hooks (Brique B) sont actifs immédiatement après génération, sans étape manuelle supplémentaire.
- [ ] Relancer la commande sur un dossier déjà initialisé ne détruit aucune donnée existante sans confirmation.

## 5. Brique B — Hooks Git

### 5.1 Objectif

Faire respecter mécaniquement les règles de traçabilité du Playbook (§1) et d'hygiène du dépôt (§12), pour qu'un commit non conforme soit bloqué avant d'exister, plutôt que corrigé après coup.

### 5.2 Hook `commit-msg`

Vérifie que chaque message de commit respecte la convention *Conventional Commits* définie au §1 du Playbook :

- Préfixe obligatoire parmi : `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `style:`, `perf:`, `test:`, `release:`.
- Rejet du commit avec message d'erreur explicite si le préfixe est absent ou invalide, citant la liste des préfixes valides.
- Avertissement (non bloquant) si le message fait moins de 10 caractères après le préfixe, pour décourager les messages trop vagues (`fix: bug`).

### 5.3 Hook `pre-commit`

Vérifie, dans l'ordre :

1. **Absence de secrets en clair** : recherche de motifs typiques (clés API, tokens, chaînes de connexion) dans les fichiers stagés. Bloquant.
2. **Absence de fichiers de debug non voulus** : rejet si des fichiers correspondant à des motifs comme `debug_*`, `test_*` (hors dossier `tests/` officiel) sont inclus dans le commit, conformément au §6 et §12 du Playbook. Bloquant, avec message indiquant le fichier concerné.
3. **Lint / format** : exécution du linter et du formatteur configurés pour le projet (définis dans `PROJECT_CONFIG.md`). Bloquant si le linter échoue ; le formatteur peut s'exécuter en mode correctif automatique si configuré ainsi.
4. **Taille des fichiers modifiés** : avertissement (non bloquant) si un fichier dépasse les seuils indicatifs du §4 du Playbook (500 lignes), pour inciter à découper plutôt que d'imposer un blocage strict qui casserait des cas légitimes.

### 5.4 Hook `pre-push`

- Rejet si la branche cible est `main` en push direct (hors cas de merge légitime), conformément au §13 du Playbook (« on ne travaille jamais directement sur `main` »).
- Exécution optionnelle des tests locaux avant push si un runner de tests est configuré, avec possibilité de bypass explicite documenté (ex. `--no-verify`) pour les cas d'urgence, mais jamais silencieux.

### 5.5 Distribution des hooks

Les hooks doivent être versionnés dans GEF (pas seulement générés localement), afin qu'une mise à jour du framework se propage aux projets existants via une commande de mise à jour explicite (ex. `gef update-hooks`), plutôt que de rester figés dans le `.git/hooks/` local de chaque projet.

### 5.6 Critères d'acceptation

- [ ] Un commit avec un message sans préfixe valide est rejeté avec un message d'erreur listant les préfixes acceptés.
- [ ] Un commit contenant une clé API en clair est bloqué.
- [ ] Un commit contenant un fichier `debug_test.py` hors du dossier `tests/` est bloqué.
- [ ] Un push direct sur `main` est bloqué avec un message renvoyant vers le workflow de branches du Playbook §13.
- [ ] Les hooks fonctionnent après un simple `gef new` ou `gef install-hooks`, sans configuration manuelle supplémentaire.

## 6. Brique C — Template CI/CD

### 6.1 Objectif

Fournir un template GitHub Actions (ou équivalent) unique, réutilisable tel quel dans chaque projet, qui applique automatiquement la séquence de contrôle définie au §9 du Playbook.

### 6.2 Séquence obligatoire

```
Push / Pull Request
   ↓
Build
   ↓
Lint
   ↓
Tests
   ↓
Analyse de sécurité (secret scanning + dépendances vulnérables)
   ↓
Coverage (rapport, seuil défini dans PROJECT_CONFIG.md)
   ↓
Release automatique (uniquement sur merge vers main, si un tag est présent)
```

### 6.3 Contraintes

- Le template doit lire ses paramètres variables (commande de build, commande de test, seuil de coverage) depuis `PROJECT_CONFIG.md` ou des variables d'environnement définies au niveau du dépôt, **pas** en dur dans le fichier de workflow, pour rester générique.
- Une étape qui échoue doit produire un résumé lisible identifiant la cause (ex. « Lint : 3 erreurs dans src/api/routes.py »), pas seulement un statut rouge.
- Le déclenchement de la release automatique doit être conditionné à l'existence d'un tag conforme au format `vX.Y.Z` défini au §11 du Playbook, jamais automatique sur un simple merge.
- Prévoir un point d'extension clair (ex. un fichier `ci-extra.yml` optionnel) pour les étapes spécifiques à un projet (ex. déploiement vers une infrastructure particulière), sans modifier le template central.

### 6.4 Critères d'acceptation

- [ ] Le template s'exécute sans modification sur un projet fraîchement généré par la Brique A.
- [ ] Une régression de lint, de test, ou de coverage fait échouer le pipeline avec un message localisant le problème.
- [ ] Aucune release n'est créée en l'absence d'un tag `vX.Y.Z` explicite.
- [ ] Le template ne contient aucune référence à un projet, un cloud provider, ou un client spécifique.

## 7. Brique D — Bibliothèque de prompts IA

### 7.1 Objectif

Garantir qu'une IA qui intervient sur un projet applique systématiquement les règles du Playbook, sans que l'utilisateur ait à les rappeler à chaque session.

### 7.2 Prompts à produire

| Fichier | Usage |
|---|---|
| `prompts/system_prompt.md` | Prompt de base à charger en début de session : rappelle les règles de traçabilité, de documentation et de méthodologie pas-à-pas (§1, §2, §3 du Playbook), et demande à l'IA d'identifier la phase du cycle de vie en cours (§0) avant d'agir. |
| `prompts/feature_development.md` | Prompt pour le développement d'une nouvelle fonctionnalité : impose la création de branche (§13), les micro-commits (§1), la mise à jour conditionnelle du README (§2), et le passage par la séquence de revue avant merge (§7). |
| `prompts/code_review.md` | Prompt pour une revue de code : checklist reprenant exactement les points du §7 du Playbook (lint, build, tests, revue, documentation, changelog). |
| `prompts/bugfix.md` | Prompt pour une correction de bug : impose la création d'une entrée `RESEARCH_LOG.md` avant de considérer le bug résolu (§6). |
| `prompts/adr_writing.md` | Prompt pour la rédaction d'un ADR : impose le format contexte / options / décision / conséquences, et le nommage `ADR-XXX-titre.md` dans `docs/adr/` (§6). |
| `prompts/new_project_kickoff.md` | Prompt à utiliser en tout début de projet : pose explicitement la question de la phase R&D vs contractuelle (§10) et aide à choisir entre les options de séparation de dépôts avant le premier commit. |

### 7.3 Contraintes de rédaction des prompts

- Chaque prompt doit citer explicitement le(s) numéro(s) de section du Playbook auxquels il correspond, pour que toute mise à jour du Playbook permette de repérer facilement les prompts à revoir.
- Aucun prompt ne doit reformuler une règle du Playbook de façon divergente — il doit y renvoyer ou la citer fidèlement.
- Les prompts doivent rester utilisables tels quels par simple copier-coller dans n'importe quel outil IA, sans dépendance à une plateforme particulière.

### 7.4 Critères d'acceptation

- [ ] Chaque prompt de la table ci-dessus existe comme fichier indépendant.
- [ ] Chaque prompt référence explicitement les sections du Playbook qu'il applique.
- [ ] `system_prompt.md` seul suffit à faire respecter les règles de traçabilité de base (§1, §2, §3) sans les autres prompts.

## 9. Ordre d'implémentation recommandé

1. **Brique D (prompts)** — le coût de mise en œuvre est le plus faible et la valeur immédiate est déjà présente sans aucun outillage technique.
2. **Brique B (hooks)** — impose mécaniquement les règles les plus fréquemment violées en pratique (messages de commit, secrets, fichiers de debug).
3. **Brique A (générateur)** — automatise la mise en place initiale, qui devient ensuite triviale une fois les hooks et templates prêts à être copiés.
4. **Brique C (CI/CD)** — dépend des choix de langage/outils faits dans la Brique A pour être pleinement paramétrable.

Cet ordre n'est pas obligatoire mais évite de construire un générateur avant de savoir précisément ce qu'il doit générer.

## 10. Critères d'acceptation globaux

Le framework est considéré complet lorsque le scénario suivant fonctionne de bout en bout, sans intervention manuelle autre que celles explicitement prévues :

1. Exécution de la commande de génération sur un nouveau dossier → structure conforme créée.
2. Une tentative de commit avec un message non conforme est rejetée.
3. Une tentative de commit avec un secret en clair est rejetée.
4. Un commit valide sur une branche `feat/...` passe sans friction.
5. Une tentative de push direct sur `main` est rejetée.
6. Une pull request vers `main` déclenche automatiquement build, lint, tests, sécurité et coverage.
7. Un tag `vX.Y.Z` mergé sur `main` déclenche une release automatique avec changelog.
8. Le prompt `system_prompt.md`, chargé dans une session IA sur ce projet généré, suffit à faire respecter la méthodologie pas-à-pas et la traçabilité sans rappel supplémentaire de l'utilisateur.
